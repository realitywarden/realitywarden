## Core Positioning

AI should not control physical devices directly.

Open Reality Studio is a simulation-first desktop runtime for Physical AI workflows. It lets AI understand a goal, check device capability, judge safety risk, run a simulation, and produce an audit trail before any future real device adapter boundary.

现在 AI 很会聊天，但如果它未来要操作真实设备，不能直接一句话就让机器动。

我最近在做一个桌面原型：Open Reality Studio。

它的想法很简单：

AI 收到自然语言指令后，不是马上控制设备，而是先：

1. 理解你想做什么
2. 检查设备有没有这个能力
3. 判断动作有没有风险
4. 先在虚拟实验室里跑一遍
5. 把执行过程和拦截原因记录下来

“把红方块放到后侧安全区”可以进入仿真执行。

“把红方块扔出桌面”会在执行前被安全系统拦截。

当前版本是 Public Alpha，只做仿真，不控制真实设备。

但我觉得 AI 进入现实世界之前，需要这样一层安全运行时。

## First Reply Templates

### If someone asks whether it controls real devices

Not yet. The current Public Alpha is simulation-only. The goal is to prove the runtime flow first: natural language -> capability check -> safety decision -> simulation -> audit. Real device execution will require a local adapter boundary and explicit safety controls.

### If someone says it is just a robot arm demo

The robot arm is the current golden path demo, but the core work is the runtime layer: device capability contracts, world model, safety decision, TaskDSL, adapter boundary, replay, and lab report. The current Public Alpha intentionally keeps real hardware disabled.

### If someone asks what is next

Next is v0.2 Foundation: visible autonomy console, Open Reality Protocol v0.1, Adapter SDK boundary, better lab report / audit, and developer onboarding docs. Real device execution stays disabled until the local runtime and safety boundary are stricter.
